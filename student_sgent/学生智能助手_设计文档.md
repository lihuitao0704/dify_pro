# 学生智能助手模块 — 系统设计文档

> 版本: 1.0.0 | 日期: 2026-07-11 | 数据库: hambaki_3

---

## 一、概述

### 1.1 模块定位

学生智能助手是教育服务系统的核心子模块，负责留学生从入学到毕业全生命周期的数字化管理。覆盖 **会话交互、心理关怀、投诉反馈、学业管理、留学追踪** 五大业务域。

### 1.2 与其他模块的关系

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ 客户研判  │     │ 客服Agent │     │ 企业助手  │
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     └────────┬───────┴───────┬────────┘
              │               │
        ┌─────▼─────┐   ┌─────▼─────┐
        │ test.students │   │ Dify/LLM │  ← 共享层
        └─────┬─────┘   └─────┬─────┘
              │               │
     ┌────────▼───────────────▼────────┐
     │         学生智能助手模块         │
     │  hambaki_3 (自有库，9张表)      │
     │  FastAPI + SQLAlchemy + MySQL   │
     └────────────────────────────────┘
```

- **test.students**: 团队共享学生表，本模块通过 `student_id` 做业务关联
- **Dify/LLM**: 负责意图识别、情绪分析、RAG 检索，通过 HTTP 调用本模块 API

### 1.3 技术栈

| 组件 | 选型 |
|------|------|
| 框架 | FastAPI 0.110+ |
| ORM | SQLAlchemy 2.0+ (pymysql) |
| 数据库 | MySQL 8.0+ (InnoDB, utf8mb4) |
| 认证 | Bearer Token (中间件) |
| 日志 | Python logging (stdout) |
| LLM | OpenAI 兼容 API (可选) |

## 二、架构设计

### 2.1 分层架构

```
┌─────────────────────────────────────────┐
│  static/index.html    单页前端 (SPA)    │  表现层
├─────────────────────────────────────────┤

├─────────────────────────────────────────┤
│  services/           业务逻辑层          │  服务层
│  ├── student_service.py                 │
│  └── nl2sql_service.py                  │
├─────────────────────────────────────────┤
│  models/             数据模型层          │  持久层

│  └── student.py      9 张表 ORM 模型    │
├─────────────────────────────────────────┤
│  schemas/            Pydantic 验证层     │  契约层
│  └── student.py                         │
├─────────────────────────────────────────┤
│  config.py           配置中心            │  基础设施
│  main.py             应用入口            │
└─────────────────────────────────────────┘
```

### 2.2 认证机制

```
请求 → auth_middleware
       ├── /docs, /health, /static ... → 白名单放行
       ├── /api, /api/              ... → 精确匹配放行
       ├── OPTIONS 预检              ... → 放行
       └── /api/* 路径
              ├── 无 Authorization header → 401
              ├── Token 不匹配            → 403
              └── Token 正确              → 放行

Token 配置: API_TOKEN 环境变量 (默认 dev-token)
Token 校验: 大小写不敏感 (bearer / Bearer)
```

### 2.3 情绪分析引擎

```
用户消息 → add_message()
            ├── 有情绪数据 (LLM/Dify 提供) → 直接存储
            └── 无情绪数据 → _analyze_emotion()
                            ├── 高危词检测 (轻生/自残...) → score=15, tag=高危
                            ├── 负面词检测 (焦虑/压力...)  → score=35, tag=焦虑
                            ├── 正面词检测 (开心/offer...) → score=80, tag=积极
                            └── 无匹配                   → score=75, tag=平稳
                            │
                            ▼
                   更新 emotion_profile_snapshots
                   新增 emotion_history 条目
                   判定 risk_level (high/medium/low)
                   
                   score < 30 或 tag=高危
                   → 自动创建 risk_interventions 预警
```

## 三、数据库设计

### 3.1 表清单

| # | 表名 | 中文名 | 记录粒度 | 核心字段数 |
|---|------|--------|----------|-----------|
| 1 | `conversation_sessions` | 会话主表 | 一人多条 | 7 |
| 2 | `conversation_messages` | 消息明细表 | 一会话多条 | 11 |
| 3 | `emotion_profile_snapshots` | 心理画像表 | 一人一条 | 9 |
| 4 | `risk_interventions` | 心理预警表 | 一人多条 | 11 |
| 5 | `feedback_tickets` | 投诉工单表 | 一人多条 | 16 |
| 6 | `academic_schedules` | 学业日程表 | 一人多条 | 12 |
| 7 | `deadline_reminders` | 考务提醒表 | 一人多条 | 11 |
| 8 | `study_intentions` | 升学意向表 | 一人多条 | 12 |
| 9 | `student_applications` | 申请进度表 | 一人多条 | 13 |

### 3.2 表关系

```
conversation_sessions (1) ── session_id ── (N) conversation_messages
                                                    │
                                          source_message_id
                                                    │
                                                    ▼
conversation_messages ──────────────── risk_interventions
       │
       student_id
       │
       ├── emotion_profile_snapshots (1:1)
       ├── feedback_tickets (1:N)
       ├── academic_schedules (1:N)
       ├── deadline_reminders (1:N)
       ├── study_intentions (1:N) ── intention_id ── student_applications (1:N)
       └── student_applications (1:N)
```

> 不使用物理外键，业务关联在 Service 层做显式 JOIN 查询。

### 3.3 关键字段说明

#### conversation_sessions

| 字段 | 类型 | 说明 |
|------|------|------|
| `message_count` | INT | 冗余计数器，`COALESCE(message_count,0)+1` 原子更新，避免 COUNT 查询 |
| `last_message_time` | DATETIME | 与 `message_count` 同事务原子更新 |

#### risk_interventions

| 字段 | 类型 | 说明 |
|------|------|------|
| `risk_tags` | JSON | 风险标签数组，如 `["失眠","学业压力","人际关系"]` |
| `resolved_time` | DATETIME | 状态变为 resolved/dismissed 时自动写入 |

#### feedback_tickets

| 字段 | 类型 | 说明 |
|------|------|------|
| `resolved_time` | DATETIME | 状态变为 resolved/closed 时自动写入，用于统计工单解决时效 |

### 3.4 时间字段策略

| 字段类型 | Python 侧 | MySQL 侧 | 目的 |
|----------|-----------|----------|------|
| `create_time` | `default=datetime.utcnow` | `server_default=CURRENT_TIMESTAMP` | flush后立即可读 + DB兜底 |
| `update_time` | `default+onupdate=datetime.utcnow` | `server_default+server_onupdate=CURRENT_TIMESTAMP` | 双重保障 |
| 业务时间 (`last_message_time`, `resolved_time`) | 服务端 `datetime.utcnow()` 写入 | — | 应用逻辑控制 |

## 四、安全设计

| 维度 | 措施 |
|------|------|
| 认证 | Bearer Token 中间件，环境变量注入，大小写不敏感 |
| CORS | 白名单来源，环境变量配置 |
| SQL 注入 | NL2SQL 仅允许单条 SELECT，多语句直接拒绝，危险关键字黑名单，1000行上限 |
| 密码 | 禁止硬编码，环境变量强制注入，启动检查弱密码告警 |
| 日志 | 创建/更新操作记录审计日志，不泄露敏感信息 |

## 五、启动方式

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 配置环境变量 (复制 .env 文件或 export)
#    DB_HOST, DB_PORT, DB_USER, DB_PASSWORD, DB_NAME
#    API_TOKEN (生产环境必改)

# 3. 建表
python create_tables.py

# 4. 初始化测试数据 (可选)
python seed_data.py

# 5. 启动
uvicorn main:app --host 0.0.0.0 --port 8008

# 6. 访问
#    前端: http://localhost:8008
#    Swagger: http://localhost:8008/docs
#    Token: dev-token
```
